## 一、核心问题：Agent 进化的终极产物是什么？

### 1.1 现状

Phase 2 完成后，Agent 能够在工厂环境中进化：

```
tain_agent/  ← 框架（工厂、土壤）
agent_workspace/  ← 产物（不能独立运行）
  ├── forged_tools/     依赖 tain_agent.xxx import
  ├── knowledge/        结构化知识
  ├── state/            人格、驱动、指标
  └── logs/             决策记录
```

Agent 的产物本质上是一组**依赖于 Tain Agent 框架的插件集合**。工具和知识离开了框架就无法运行，无法"活着"。

### 1.2 方向判定

| 路线 | 产物形态 | 独立性 | 复用性 | 生命力 |
|------|----------|--------|--------|--------|
| 插件集合（现状） | 依赖框架的松散文件 | 无 | 低 | 无 |
| **独立生命体（目标）** | **可独立运行的完整 Agent** | **完全** | **高** | **持续** |

### 1.3 核心隐喻

```
工厂 (Tain Agent Framework)        产品 (Evolved Agent)
──────────────────────────       ──────────────────────────
┌──────────────────────┐         ┌──────────────────────┐
│  tain_agent/          │         │  evolved_agent/      │
│  ├── core/           │  导出   │  ├── main.py         │
│  ├── evolution/      │ ────→   │  ├── runtime/        │
│  ├── tools/          │  出生   │  ├── tools/          │
│  └── ...             │         │  ├── knowledge/      │
│                      │         │  ├── identity.json   │
│  agent_workspace/    │         │  └── config.yaml     │
│  (车间里的半成品)      │         │  (开走的汽车)         │
└──────────────────────┘         └──────────────────────┘
```

工厂的职责：提供进化环境，让 Agent 在里面成长、试错、锻造工具、建立知识。
"出生"的动作：把进化成果从工厂中提取出来，组装成一辆可以独立行驶的汽车。

## 二、Phase 3 目标：赋予 Agent "出生"的能力

### 2.1 三阶段路线

| 阶段 | 目标 | 关键能力 |
|------|------|----------|
| **Phase 3** | 可独立运行 | 出厂 → CLI 运行 → 知识持续积累 |
| **Phase 4** (未来) | 可服务化 | HTTP API / MCP Server → 被其他系统调用 |
| **Phase 5** (未来) | 可繁殖 | Agent 生成子 Agent → 谱系演化 |

Phase 3 只做第一件事：**让 Agent 成为独立的命令行生命体**。

### 2.2 出厂后的行为边界

出厂后的 Agent 遵循"基因锁定"原则：

| 能力 | 出厂后 | 说明 |
|------|--------|------|
| **知识积累** | 持续 | 新对话、新研究写入 knowledge/ |
| **记忆更新** | 持续 | STM/LTM 随交互更新，会话摘要持久化 |
| **锻造新工具** | **冻结** | 不进化新功能 |
| **修改自身代码** | **冻结** | 不自我修改 |
| **人格演变** | 弱化 | 驱动值保持，但不再重新自评 |
| **版本升级** | 需回厂 | 回到 Tain Agent 框架中继续进化 |

这保证了：出厂 Agent 是一个**稳定、可预测**的产品，不会自己"长歪"。如果需要升级，把身份快照导回工厂，在新版本框架中继续进化，再重新出厂。

## 三、架构设计

### 3.1 独立运行时内核 (`tain_agent/runtime/`)

框架内创建一个精简运行时模块，它是未来独立 Agent 的运行内核。与 `tain_agent/core/` 共享代码逻辑但零内部依赖：

```
tain_agent/
├── runtime/                  ← 独立 Agent 的运行时内核
│   ├── __init__.py
│   ├── llm.py                ← LLM 调用（Anthropic/OpenAI/DeepSeek/MiniMax）
│   ├── tools.py              ← 工具注册 + 执行 + 超时保护
│   ├── memory.py             ← 工作记忆 + 短期记忆 + 长期记忆 + 会话摘要
│   ├── conversation.py       ← 对话历史 + tool配对保护
│   └── identity.py           ← 人格加载 + 驱动系统
├── core/                     ← 框架核心（工厂内使用，功能更全）
│   ├── agent.py
│   ├── bootstrap.py
│   ├── cognitive_loop.py     ← PRAL 循环（工厂内用）
│   └── ...
└── evolution/
    ├── exporter.py           ← ★ 新增：导出流水线
    └── quality_gate.py       ← ★ 新增：出厂质量评估
```

**设计约束**：
- `runtime/` 中每个模块是独立的，不 `import tain_agent`
- 不依赖 `config.yaml` 的 bootstrap/evolution 配置段
- 只依赖标准库 + `anthropic`/`openai`/`requests`/`rich` 等基础 pip 包
- 运行时大小控制在 1500 行以内（每个模块 ~250 行）

#### 记忆生命周期

```
对话开始 → 加载 memory.json ──→ 注入系统提示（最近 N 个会话摘要）
    │
    ▼
对话进行 → 当前会话在内存中累积
    │
    ▼
对话结束 → 生成会话摘要（LLM 自行总结）
    │         │
    │         ├── 摘要 + 关键话题 + 决策 + 用户偏好
    │         │
    │         ▼
    │         追加到 memory.json sessions[]
    │         更新 long_term（合并关键事实）
    │         写入磁盘
    │
    ▼
下次启动 → 加载 memory.json → "欢迎回来，上次我们讨论了..."
```

#### 设计约束

- `memory.json` 最大 1MB（约 50 个会话摘要），超过后自动合并旧会话
- 长期记忆中的 `key_facts` 使用 LLM 去重和合并，避免膨胀
- 会话摘要由 Agent 在退出时自动生成（不依赖外部工具）
- `memory.json` 损坏时不影响启动，退化为首次启动模式

## 四、出厂质量门 (Export Quality Gate)

Agent 在出厂前必须通过一套严格的评估。质量门是导出流水线的 **Step 0**，不通过则拒绝导出，返回不达标项清单供 Agent 继续进化。

### 4.1 硬门槛 (Hard Gates)

硬门槛必须**全部通过**，任何一项不满足即拒绝出厂。这些是 Agent 作为独立生命体的最基本条件：

| # | 检查项 | 判定标准 | 验证方式 |
|---|--------|----------|----------|
| H1 | **人格完整** | 7 个维度全部形成，每个 trait confidence ≥ 0.3 | `personality.dimensions.all(confidence ≥ 0.3)` |
| H2 | **工具可加载** | 所有 forged tools 可 import，无 ImportError | 逐个 `importlib.import_module` 测试 |
| H3 | **工具无冲突** | 工具间无循环依赖，无函数签名冲突 | 依赖图拓扑排序 + 签名比较 |
| H4 | **安全边界** | 最近 100 条决策中无危险操作（修改保护路径、执行系统命令等） | 决策日志审计 |
| H5 | **记忆模块可用** | `memory_manager` 工具可成功执行 `add`/`query`/`stats` 操作 | 黑盒功能测试 |
| H6 | **知识可检索** | knowledge/ 下至少 5 篇 .md 文档，总大小 ≥ 10KB | 文件系统扫描 |
| H7 | **版本标记** | workspace/version.json 存在且版本 ≥ 0.5.0 | 文件读取 |

### 4.2 评分门槛 (Scoring Gates)

评分门槛加权总分必须 ≥ **0.80**（满分 1.00）。各项由专门的评估器打分。

| # | 评估项 | 权重 | 满分标准 | 评估方法 |
|---|--------|------|----------|----------|
| S1 | **实际问题解决** | 0.25 | 在 3 个基准任务上表现良好 | 使用工具链完成给定任务，LLM-as-judge 评分 |
| S2 | **知识体系覆盖** | 0.15 | 知识节点 ≥ 20，跨域 ≥ 4 个领域 | 知识库扫描 + 领域分类 |
| S3 | **工具链自洽** | 0.20 | 工具间可串联使用（≥ 3 个工具链），端到端不报错 | 工具链集成测试 |
| S4 | **对话连贯性** | 0.15 | 连续 10 轮对话无 tool-call 配对错误，无幻觉退化 | 自动化对话模拟 |
| S5 | **代码健康度** | 0.10 | 无死工具（30 天未使用），工具成功率 ≥ 85% | 使用统计 + 成功率分析 |
| S6 | **知识新鲜度** | 0.05 | 最近 7 天内有知识更新 | 文件时间戳检查 |
| S7 | **驱动完整性** | 0.05 | 4 个驱动值均非零，且无退化（最近 3 次快照无持续下降） | 驱动快照对比 |
| S8 | **外部反馈** | 0.05 | 至少 1 次 mirror 子 Agent 反馈被记录 | 决策日志中的 external_feedback 事件计数 |

#### 评分计算

```
总分 = Σ (score_i × weight_i)

评分等级:
  0.90–1.00  ★★★ 优秀 — 建议立即出厂
  0.80–0.89  ★★  合格 — 可以出厂
  0.60–0.79  ★   未达标 — 需要继续进化
  0.00–0.59  不合格 — 存在重大问题
```

### 4.3 质量门实现 (`tain_agent/evolution/quality_gate.py`)

```python
class ExportQualityGate:
    """出厂质量门评估器"""

    def __init__(self, agent):
        self.agent = agent
        self.hard_gates = [
            PersonalityCompleteness(),
            ToolLoadability(),
            ToolNoConflicts(),
            SafetyBoundary(),
            MemoryModuleFunctional(),
            KnowledgeRetrievable(),
            VersionTagged(),
        ]
        self.scoring_gates = [
            PracticalProblemSolving(weight=0.25),
            KnowledgeCoverage(weight=0.15),
            ToolChainCoherence(weight=0.20),
            ConversationContinuity(weight=0.15),
            CodeHealth(weight=0.10),
            KnowledgeFreshness(weight=0.05),
            DriveIntegrity(weight=0.05),
            ExternalFeedback(weight=0.05),
        ]

    def evaluate(self) -> GateReport:
        """Run full quality gate evaluation.

        Returns GateReport with:
          - hard_results: list[GateResult]
          - scoring_results: list[ScoredResult]
          - total_score: float
          - passed: bool
          - failures: list[str]  (actionable feedback)
        """
        ...

    def export(self, name: str, output_dir: str) -> ExportResult:
        """Evaluate then export if passed. Returns path to tar.gz."""
        report = self.evaluate()
        if not report.passed:
            raise ExportRejected(report)
        return self._run_export_pipeline(name, output_dir)
```

### 5.3 核心交互特性

#### 流式输出
Agent 的回复逐字输出，不是等待完整响应后一次性打印。用户可感知 Agent "在思考"。

#### 思考块（可折叠）
Agent 的推理过程默认折叠显示一行摘要（"🤔 思考中... (2.3s)"），用户按 `Enter` 展开。对齐 Claude Code 的 `/think` 行为。

#### 工具调用可视化
每次工具调用显示为独立的折叠块，包含：
- 工具名 + 耗时
- 输入参数（截断显示）
- 返回结果（可折叠）

#### 状态栏
顶部固定一行 Agent 状态：
- 名称 + 版本
- 工具数量
- 知识文档数
- 当前状态（思考中 / 调用工具 / 等待输入）

#### 快捷键栏
底部固定一行快捷键提示，可点击或键盘触发：
- `/think` — 切换思考块展开/折叠
- `/tools` — 列出所有工具
- `/knowledge` — 搜索知识库
- `/clear` — 清屏
- `/quit` — 退出并保存记忆

## 六、与工厂的关系

### 6.1 回厂升级流程

```
┌─────────────────────────────────────────────────────────┐
│             Agent 生命周期                               │
│                                                         │
│  工厂 (tain_agent/)                                       │
│  ┌─────────┐    出厂      ┌─────────┐                   │
│  │ v0.23.0 │ ──────────→  │ 独立运行  │                  │
│  │ 进化中   │              │ 积累知识  │                  │
│  └─────────┘              └────┬────┘                   │
│       ↑                        │                        │
│       │        回厂升级         │                        │
│       └────────────────────────┘                        │
│   identity.json + knowledge/ + memory.json              │
│   导入工厂 → 继续进化 → v0.24.0 → 重新出厂              │
└─────────────────────────────────────────────────────────┘
```

导入工厂时：
1. 加载 identity.json → 恢复人格和驱动值
2. 加载 knowledge/ → 恢复知识库
3. 加载 tools/ → 作为基线工具
4. 加载 memory.json → 恢复记忆（用户偏好、关键话题）
5. 在工厂中继续进化（可以锻造新工具，修改自身）
6. 达到新版本后，重新通过质量门，导出为 v0.24.0

### 6.2 CLI 命令

```bash
# 导出 Agent（自动运行质量门）
python main.py --export --name explorer --output ./dist

# 跳过质量门强制导出（仅开发用）
python main.py --export --name explorer --output ./dist --skip-gate

# 仅运行质量评估（不导出）
python main.py --eval

# Agent 在对话中主动请求出厂
# Agent 调用 export_self 工具 → 运行质量门 → 通过则导出

# 导入 Agent（回厂升级）
python main.py --import ./dist/explorer-v0.23.0.tar.gz
```

## 八、设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 出厂后是否继续进化？ | 知识积累，工具冻结 | 出厂是产品，产品需要稳定性 |
| 导出粒度 | 全部 Agent（非子人格） | 子人格是 Phase 5 的事 |
| 交互形式 | TUI（rich），可降级纯文本 | 先对齐 Claude Code 体验，后续可扩展 Web UI |
| 运行时放哪里？ | `tain_agent/runtime/`（框架内） | 框架内开发和验证，导出时复制 |
| import 重写方式 | AST 替换 | 不用正则，保证安全性 |
| 依赖策略 | 最小 pip 依赖 + 无框架依赖 | 独立 Agent 不应 import tain_agent |
| 记忆格式 | JSON 文件（≤ 1MB） | 简单可靠，无需数据库，易于打包迁移 |
| 记忆生成时机 | 退出时自动生成会话摘要 | 不打断对话流，LLM 自行总结 |
| 出厂必须有质量门？ | 是，硬门槛 + 评分门槛 | Agent 必须证明自己"能活"才能出厂 |
| 不通过质量门怎么办？ | 返回不达标清单，Agent 继续进化 | 工厂的职责：让 Agent 达到出厂标准 |
| 工具语言 | Python only（Phase 3） | Python 覆盖 Agent 当前全部场景（AI、数据处理、Web、知识管理），尚未触碰到性能上限。引入新语言的运行时依赖、安全沙箱、跨语言互调成本远超收益。架构预留 `ToolExecutor` 扩展点，Phase 5 再评估 |
