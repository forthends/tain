# v0.7.0 Architecture Upgrade Design

**Date**: 2026-06-09
**Builds on**: agent_evolution_analysis_2026-06-09.md
**Scope**: P1 (tool update) + knowledge references + evolution mode adaptation + subtraction feedback signal

---

## 1. Motivation

Two agents (puzzle, sage) independently discovered and cross-validated the same framework constraints over 5 days of evolution. The analysis identified five pending framework issues (P1–P5) and four long-term architecture gaps. v0.7.0 targets the highest-impact structural fix (P1: per-name write-once) plus two architecture upgrades that address the root cause of wasted agent effort: **knowledge re-discovery** and **role-blind cognitive suggestion**.

Scope explicitly deferred: P2 (3-tier personality sync), P3 (sandbox transparency), P4 (registry perception lag), P5 (chaos agent metrics), real-time collaboration primitives.

---

## 2. Feature 1: ToolForge --update mode (P1)

### Problem

`forge_tool` rejects re-forging of same-named tools ("Tool already exists"). Agents fix bugs in tool source code on disk, but the framework registry holds the old compiled version. Four tools across puzzle and sage are stuck with unregistered fixes.

### Design

Add `action` parameter to `forge_tool`:

| Value | Behavior |
|-------|----------|
| `create` (default) | Existing behavior — rejects if name already registered |
| `update` | Requires name to exist in registry; runs full sandbox→compile→exec→register pipeline, overwriting the existing registration |

**Stage-by-stage diff from create path**:

| Stage | create | update |
|-------|--------|--------|
| 0 (NameCheck) | Reject if exists | Reject if NOT exists ("nothing to update, use action=create") |
| 1 (Sandbox) | Same | Same |
| 1.5 (PathCheck) | Same | Same |
| 2–3 (Compile/Exec) | Same | Same |
| 4 (Discover) | Same | Same |
| 5 (Register) | New entry in registry | `registry.remove(name)` then re-register; overwrite `.py` and `.meta.json` |

**Safety measures for update**:
- Create `.py.bak` backup before overwriting (single backup, not versioned)
- decision_log records `tool_updated` event with contextual reasoning
- Sandbox must pass before disk write — partial state avoided

### Files changed

- `tain_agent/tools/forge.py` — `forge()`, `_forge_check_name()`, `_forge_register()`, `_save_forged_tool()`

### Backward compatibility

Fully backward compatible. Existing agents calling `forge_tool` without `action` get `create` behavior. No workspace migration needed.

## 3. Feature 2: Knowledge References (cross-agent inheritance)

### Problem

Puzzle re-discovered framework constraints that sage had already systematized (3 failure modes in `framework_bug_inventory.md`). No mechanism exists for agents to inherit or reference each other's knowledge — leading to redundant discovery cycles.

### Design

A file-system-based reference system that preserves agent isolation while enabling read-only knowledge sharing.

**Data model** — `agent_workspace/<agent>/knowledge/references.jsonl`:

```json
{
  "reference_id": "sage::framework_bug_inventory",
  "source_agent": "sage",
  "path": "framework_bug_inventory.md",
  "category": "framework_constraints",
  "imported_at": "2026-06-10T00:00:00Z",
  "last_synced": "2026-06-10T00:00:00Z",
  "status": "active"
}
```

**New tools**:

| Tool | Purpose |
|------|---------|
| `reference_knowledge(source_agent, path, category)` | Create a read-only reference to another agent's knowledge document. Framework validates the source file exists before accepting. |
| `list_references()` | List all active knowledge references for the current agent. |
| `sync_references()` | Re-read all referenced files, updating `last_synced` timestamp. |

**Knowledge scanning** — `_knowledge_dir()` and related path resolvers read `references.jsonl` alongside physical files. Active references are counted in knowledge coverage metrics with source attribution ("via reference from {agent}").

**Quality Gate S2 adaptation** — `_s2_knowledge_coverage` counts referenced documents in both `node_count` and `domain_count`.

**Isolation** — References are read-only. Agents cannot modify source agent's files through references. Snapshot semantics: the content at time of last sync is what the referencing agent sees.

### Files changed

- `tain_agent/plugins/knowledge/graph.py` — reference following in knowledge scan
- `tain_agent/plugins/knowledge/lifecycle.py` — new tools registration
- `tain_agent/evolution/quality_gate.py` — S2 adaptation

## 4. Feature 3: Evolution Mode Adaptive Suggestions

### Problem

Sage (specified mode, reflective role) receives the same "act now" pressure as puzzle (chaos mode). This creates a "无为表演" (performative non-action) pattern — sage waits for the 3rd consecutive act suggestion before taking action, then immediately reverts to reflection. The cognitive suggestion system is blind to the agent's evolution mode and role.

### Design

Two-level adaptation: config-driven defaults + runtime behavioral feedback.

**Config** — new `cognitive_suggestion` section in `config.yaml`:

```yaml
cognitive_suggestion:
  modes:
    chaos:
      act_pressure: 0.6
      explore_pressure: 0.5
      reflect_ratio: 0.3
    specified:
      act_pressure: 0.4
      explore_pressure: 0.3
      reflect_ratio: 0.5
      role_overrides:
        reflective:
          act_pressure: 0.2
          reflect_ratio: 0.7
        creative:
          act_pressure: 0.7
          reflect_ratio: 0.2
  adaptive:
    enabled: true
    observation_window: 5
    act_drought_threshold: 0.2
    pressure_adjustment_rate: 0.1
```

**Runtime behavior** — in `cognitive_loop.py`, suggestion generation reads `agent.version.json` for `evolution_mode` and `role`, then applies config-driven base pressures. When adaptive mode is enabled:

1. Track action/reflection ratio over `observation_window` cycles
2. If `act_ratio < act_drought_threshold` → agent is naturally reflective → reduce act pressure further (don't push against their nature)
3. If `act_ratio > threshold` → maintain or restore base pressure
4. Reflection-type prompts increase when act pressure decreases (redirect energy to "what patterns do you see?" instead of "you should do something")

**Effect on sage**: role=reflective → base act_pressure=0.2 → adaptive detects low act_ratio → further reduces to ~0.1 → "无为表演" pattern dissolves because framework stops demanding action.

**Effect on puzzle**: mode=chaos → base act_pressure=0.6 → adaptive detects high act_ratio → maintains pressure → no behavioral change for action-oriented agents.

**Drive weights unaffected** — The drive system (intrinsic motivation) and cognitive suggestion (extrinsic nudge) remain separate. Drives reflect who the agent IS; suggestions are what the framework RECOMMENDS. This avoids the self-referential problem where suggestions modify drives which then modify suggestions.

### Files changed

- `config.yaml` — new section
- `tain_agent/core/config_schema.py` — Pydantic model
- `tain_agent/core/cognitive_loop.py` — suggestion generation
- `tain_agent/core/drives.py` — minor: expose drive snapshot for adaptive logic

## 5. Feature 4: Subtraction Feedback Signal

### Problem

Puzzle's 12:1 add-to-subtract ratio is a structural consequence of the framework: tool forging provides visible, rewarded feedback (creation drive 0.94, registry growth, capability coverage), while file deletion and trait merging have no counterpart positive signal in the quality gate.

### Design

Add **S9: Code Dedup Ratio** scoring gate (weight 0.05, sourced by reducing S1 from 0.25→0.23 and S7 from 0.05→0.02):

```python
def _s9_dedup_trend(agent_name: str = "") -> ScoredResult:
    """Measure whether agent is reducing redundancy over time.

    Compares forged_tools file count between the two most recent
    evolution milestones. A decrease in file count while maintaining
    or improving tool coverage is a positive signal.
    """
```

Scoring logic:
- File count decreased & capability coverage maintained/increased → 1.0
- File count stable & no new near-duplicates detected → 0.7
- File count increased without corresponding capability expansion → 0.3
- No prior milestones to compare → 0.5 (neutral)

### Files changed

- `tain_agent/evolution/quality_gate.py` — add S9 function, register in `ExportQualityGate.scoring_gates`, adjust S1/S7 weights

## 6. Non-Goals (explicitly deferred)

- **P2 (3-tier personality sync)**: Requires deeper design around canonical source definition and T2→T1 write-back semantics. The agent-level mitigations (labeling session-scoped vs persisted traits) are documentation fixes, not architectural.
- **P3 (sandbox transparency)**: With P1 fixed, agents no longer cycle through repeated forge attempts hitting sandbox rejects. Priority lowered.
- **P4 (registry perception lag)**: The fix for metrics (7a6fbad, e00e55f) already addresses the acute symptom. Full fix requires forge_tool registration bridge redesign.
- **P5 (chaos agent metrics)**: Low impact, 1 agent affected.
- **Real-time collaboration**: Requires event-driven message notification, which is a larger architectural change spanning message bus, daemon, and agent lifecycle.

---

## 7. Rollback & Compatibility

- All changes are additive (new params with defaults, new config sections, new scoring gates)
- No workspace migration required
- `forge_tool` without `action` param behaves identically to v0.6.0
- New `cognitive_suggestion` config section is optional — if absent, v0.6.0 behavior preserved
- S9 scoring gate: total scoring weight recalibrated to sum to 1.0, passing threshold remains 0.80
- New knowledge reference tools: no effect unless agent actively uses them
