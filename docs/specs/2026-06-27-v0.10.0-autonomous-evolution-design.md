# v0.10.0 自主进化闭环设计

**日期：** 2026-06-27
**版本：** v0.9.0 → v0.10.0
**依据：** [洞察报告](../../report/insight-report-2026-06-26.md) 长期愿景 #10 + 中期项 #13/#15/#19
**策略：** 闭合进化循环代码生成瓶颈 + 集成测试 + 配置校验
**前置：** v0.9.0 稳定化 + 架构迁移（Mixin → Kernel/Plugin）已完成

---

## 一、动机

洞察报告指出框架核心差异化主张——"诚实进化"——存在结构性断裂：进化管线在 `design` 阶段停止，等待人类提供代码。"自主进化"实际上是"缺口检测 + 验证 + 注册流水线"。

v0.9.0 完成了稳定化，AgentKernel/Plugin 架构成为唯一入口。现在是闭合进化循环的时机——让 Agent 真正能从"发现缺口"到"自主写代码修复"。

### 1.1 当前断点

```
缺口检测 ✓ → 方案设计 ✓ → 代码生成 ✗（_code_generator hook 从未注入）
    → forge ✗（人类提供代码后手动触发）→ 验证 ✗ → 效果评估 ✗
```

`ImprovementLoop._code_generator` 和 `_forge_cycle` 定义但未注入。`ClosedForgeCycle` 已有完整的 GENERATE → FORGE → VERIFY → REGISTER 6 阶段，但只在 `_maybe_generate_fallback_code()` 中作为 fallback 调用。

### 1.2 关键澄清

- **LLM 代码生成使用 Agent 自己的 LLM backend** — 与 PRAL loop 中用于推理的是同一个 backend。不引入独立的代码生成模型。
- **在线验证的测试输入从契约的 input_schema 自动生成** — 不依赖人类编写测试用例。对于 `{"type": "string"}` 类型的参数，生成边界值（空字符串、典型值、超长字符串）进行测试。
- **维度评估器缺失时的降级策略** — 当 `code_entropy`、`knowledge_freshness` 等 forged 工具不可用时（首次启动），维度评分返回 0.0（不触发改进），而非崩溃。

### 1.3 设计原则

1. **全自动闭环** — 从缺口检测到效果评估，零人类介入
2. **安全优先** — 三层防线：sandbox → 行为契约 → 自动回滚
3. **直接替换** — 删除 ImprovementLoop，AutonomousEvolutionLoop 成为唯一路径
4. **可观测** — 每个阶段记录 lineage 事件，回滚有完整审计追踪

## 二、目标架构

```
Agent 启动 (main.py / agent_factory)
    │
    ├── PRAL Loop (kernel/pral.py)
    │   └── 认知循环：感知 → 推理 → 行动 → 学习
    │
    └── AutonomousEvolutionLoop (evolution/autonomous_loop.py) [后台线程]
        │   ┌── 独立运行，不阻塞 PRAL 认知循环
        │   ┌── 通过 Plugin 接口读取工具状态、写入新工具
        │   ┌── PRAL loop 在 cycle 边界接收进化通知（工具列表变更事件）
        │
        ├── GapDetector    → 复用现有 8 维度触发评估
        ├── SpecGenerator  → ImprovementSpec（基于缺口分析）
        ├── CodeGenerator  → LLM 生成代码 + 行为契约
        ├── BehaviorContract → AST 验证 + 越界监控
        ├── ClosedForgeCycle → sandbox 7 阶段验证（现有，不变）
        ├── OnlineVerifier → 工具部署后调用验证
        └── RollbackManager → 快照管理 + 自动回滚
```

## 三、新组件设计

### 3.1 AutonomousEvolutionLoop

**文件：** `tain_agent/evolution/autonomous_loop.py`（新建，~300 行）

替代旧的 `ImprovementLoop`。完整 8 阶段执行：

```
GAP_DETECT → SPEC_DESIGN → CODE_GENERATE → CONTRACT_CHECK
    → SANDBOX_FORGE → REGISTER → ONLINE_VERIFY → EVALUATE
```

**核心方法：**

```python
class AutonomousEvolutionLoop:
    """Closed autonomous evolution — gap to deployment, zero human touch."""

    def __init__(self, llm_backend, tool_plugin, quality_gate,
                 knowledge_plugin, memory_plugin, lineage):
        self._llm = llm_backend
        self._tools = tool_plugin
        self._gate = quality_gate
        self._knowledge = knowledge_plugin
        self._memory = memory_plugin
        self._lineage = lineage
        self._forge_cycle = ClosedForgeCycle(...)
        self._snapshots: dict[str, ToolSnapshot] = {}
        self._contracts: dict[str, BehaviorContract] = {}

        # 配置
        self.min_interval_seconds = 300
        self.max_improvements_per_session = 10
        self.max_generate_retries = 3
        self.rollback_on_failure_count = 3
        self.contract_enforcement = "strict"  # strict | warn | off

        # 运行时状态
        self._running = False
        self._paused = False
        self._improvements_this_session = 0
        self._last_cycle_at = None
        self._cycle_history: list[dict] = []

    def run_one_cycle(self) -> CycleResult:
        """Execute a complete autonomous evolution cycle."""
        # 1. GAP_DETECT — 评估是否需要改进
        assessment = self._assess_need()
        if not assessment.should_trigger:
            return CycleResult.skipped(assessment)

        # 2. SPEC_DESIGN — 生成改进规格
        spec = self._generate_spec(assessment)

        # 3. CODE_GENERATE — LLM 生成代码 + 行为契约
        code, contract = self._generate_code(spec)
        if code is None:
            return CycleResult.failed("code_generation", spec)

        # 4. CONTRACT_CHECK — AST 验证契约声明
        contract_ok = self._check_contract(code, contract)
        if not contract_ok:
            if self.contract_enforcement == "strict":
                # 重试最多 2 次
                for _ in range(2):
                    code, contract = self._generate_code(spec, retry=True)
                    if self._check_contract(code, contract):
                        break
                else:
                    return CycleResult.failed("contract_check", spec)

        # 5. SANDBOX_FORGE — 通过 sandbox 管线
        forge_result = self._forge_cycle.run(spec, code)
        if not forge_result.success:
            return CycleResult.failed("sandbox_forge", spec, forge_result)

        # 6. REGISTER — 注册工具
        self._tools.register(spec.function_name, ...)

        # 7. ONLINE_VERIFY — 部署后调用验证
        snapshot_before = self._capture_snapshot()
        verify_result = self._verify_online(spec.function_name)
        if verify_result.consecutive_failures >= self.rollback_on_failure_count:
            self._rollback(spec.function_name, snapshot_before)
            return CycleResult.failed("online_verify", spec)

        # 8. EVALUATE — 质量门 delta 对比
        quality_delta = self._evaluate_quality_delta(snapshot_before)
        if quality_delta.degraded:
            self._rollback(spec.function_name, snapshot_before)
            return CycleResult.failed("quality_degraded", spec)

        # 成功 — 保存快照，记录 lineage
        self._save_snapshot(spec.function_name, code, contract)
        self._lineage.record_evolution(spec, contract)

        return CycleResult.success(spec, code, contract)
```

### 3.2 BehaviorContract（行为契约）

**文件：** `tain_agent/evolution/behavior_contract.py`（新建，~80 行）

```python
@dataclass
class BehaviorContract:
    """工具必须声明的行为边界。LLM 生成代码时同时产出此契约。"""
    tool_name: str
    input_schema: dict       # JSON Schema of expected inputs
    output_schema: dict      # JSON Schema of expected outputs
    side_effects: list[str]  # ["file_read", "file_write", "network", "none"]
    max_runtime_ms: int      # 最大允许执行时间，默认 5000

    @classmethod
    def from_generated(cls, tool_name: str, contract_json: dict) -> "BehaviorContract":
        """Parse contract from LLM-generated JSON, with validation."""
        # 验证 side_effects 仅包含合法值
        valid_effects = {"file_read", "file_write", "network", "subprocess", "none"}
        side_effects = contract_json.get("side_effects", ["none"])
        invalid = set(side_effects) - valid_effects
        if invalid:
            raise ContractValidationError(f"Invalid side effects: {invalid}")
        return cls(
            tool_name=tool_name,
            input_schema=contract_json.get("input_schema", {}),
            output_schema=contract_json.get("output_schema", {}),
            side_effects=side_effects,
            max_runtime_ms=contract_json.get("max_runtime_ms", 5000),
        )

    def verify_code_compliance(self, code: str) -> ContractComplianceResult:
        """AST-parse the code and verify it doesn't exceed declared boundaries."""
        import ast
        tree = ast.parse(code)
        violations = []

        # 检查 import 是否匹配声明的 side_effects
        declared = set(self.side_effects)
        for node in ast.walk(tree):
            if isinstance(node, ast.Import):
                for alias in node.names:
                    mod = alias.name.split(".")[0]
                    self._check_module_compliance(mod, declared, violations)
            elif isinstance(node, ast.ImportFrom):
                if node.module:
                    mod = node.module.split(".")[0]
                    self._check_module_compliance(mod, declared, violations)

        return ContractComplianceResult(
            compliant=len(violations) == 0,
            violations=violations,
        )
```

## 六、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| LLM 生成代码质量不稳定 | 工具不可用，浪费进化配额 | Contract AST 验证 + sandbox 双重过滤 + 最多 3 次重试 |
| 自动回滚过于激进 | 误回滚正常工具 | 仅对比 S1-S4（工具执行指标），不对比 H 类硬性门；连续失败阈值可配置 |
| 旧 ImprovementLoop 有未发现引用 | 删除后 import 错误 | Phase 1 前全局 grep 确认所有引用点 |
| LLM 输出格式不一致 | 代码/契约解析失败 | 明确的结构化提示 + markdown fence 剥离 + JSON schema 验证 |
| 进化循环在后台线程运行 | 并发访问 Plugin 状态 | PRAL loop 在 cycle 边界检查进化信号，不在进化线程中写 PRAL 状态 |

## 七、设计决策记录

- **直接替换而非并存：** 旧 ImprovementLoop 的"人类参与模式"是无意的设计妥协而非有意的特性。它存在是因为代码生成从未被实现——一旦代码生成可工作，就不需要人类参与路径。
- **行为契约独立于 sandbox：** sandbox 检查"代码能跑"，契约检查"代码不会越界"。两者互为补充——sandbox 防止恶意代码，契约防止无意越权。
- **回滚仅对比 S 类门：** H 类硬性门（人格完整性、版本号等）不会被一次工具更新影响。S1-S4（工具成功率、知识覆盖、工具链一致性、动作多样性）是工具更新的直接观测窗。
- **LLM 生成提示中注入 sandbox allowlist：** 源头预防优于事后报错。LLM 知道可用模块就不会生成不可用 import，减少重试浪费。
- **集成测试放在本迭代：** 全自动闭环的正确性不能靠单元测试单独保证。完整生命周期的集成测试是正确性的必要保障。
