## 1. 背景与置信度校准

### 1.1 路线图诊断准确度

v0.8-roadmap.md（2026-06-10）基于 puzzle 和 sage 两个 Agent 在 5 天内共 58 个进化周期的观测，识别出 4 个 P0/P1 架构问题和若干 P2/P3 优化项。通过逐项与当前代码实际状态比对：

| 路线图项 | 诊断置信度 | 当前状态 |

| 0.1 forge_tool 更新通路截断 | **HIGH** | forge.py 已支持 `action="update"`（commit `0778dd0`），但 bootstrap.py:247 包裹函数未透传 `action` 参数——**半修复** |
| 0.2 多 Agent 人格状态污染 | **HIGH** | personality.py:538 仍硬编码 `_Path("agent_workspace/state")`，未使用 agent 专属路径——**未修复** |
| 1.1 三状态单向死区 | **HIGH** | T1/T2/T3 之间无任何同步机制，双 Agent 独立验证两种失同步模式——**未修复** |

**核心洞察：P0/P1 四项 HIGH 置信度，路线图诊断准确。两处"半修复"（forge 和 metrics）制造已解决的假象，需在本轮彻底闭环。**

### 1.2 从严重度排序到链路排序

原路线图按 P0/P1/P2/P3 严重度排序，但框架改动按模块边界组织更合理。本次采用 **Agent 痛点解决链路** 维度，每条链路覆盖一个 Agent 视角的端到端体验：

- **链路 A：工具自主进化闭环** — Agent 能自主完成"发现问题→了解约束→修复代码→重新注册→框架可见"的完整闭环
- **链路 B：人格生命周期管理** — Agent 人格跨 session 持久、Agent 间隔离、三视角一致
- **链路 C：元认知与协作基础设施** — Agent 低成本自省、实时通信、诊断结果反哺框架

## 2. 链路 A：工具自主进化闭环

### 2.1 当前断点

```
发现工具 bug
  → 修改磁盘上的工具源码 ✓
  → forge_tool(..., action="update") ✗  bootstrap 包裹函数未暴露 action 参数
  → 知道沙箱允许哪些 import ✗  白名单不可见，只能试错
  → assess_capabilities 反映更新 ✗  metrics 路径不一致，工具可见性滞后
```

### 2.2 目标体验

```
发现工具 bug
  → get_sandbox_allowlist() → 确认可用模块
  → forge_tool(name, code, action="update") → 一步重新注册，框架注册表同步更新
  → evolution_metrics / assess_capabilities → 立即反映更新后的工具状态
```

### 2.3 具体改动

#### A-1：bootstrap.py — forge_tool 包裹函数增加 action 参数

**文件：** `tain_agent/core/bootstrap.py`（`_register_forge` 方法，约第 234-269 行）

**当前：**
```python
def forge_tool(name: str, description: str, code: str,
               parameters: str = "{}", dependencies: str = "[]") -> str:
    ...
    result = self.a.forge.forge(name, description, code, params)
```

**改为：**
```python
def forge_tool(name: str, description: str, code: str,
               parameters: str = "{}", dependencies: str = "[]",
               action: str = "create") -> str:
    ...
    if action not in ("create", "update"):
        return json.dumps({"success": False, "error": f"Invalid action: {action}. Use 'create' or 'update'."})
    result = self.a.forge.forge(name, description, code, params, action=action)
```

工具描述同步更新，增加 `action` 参数声明（`"create"` 锻造新工具，`"update"` 更新已有工具）。

- forge_tool 和 execute_code 共享同一份白名单

**新增函数：** `get_sandbox_allowlist()` → 返回当前允许的模块列表及简要说明

**文件：** `tain_agent/core/bootstrap.py`

- 注册 `get_sandbox_allowlist` 为框架内置工具

#### A-3：reporter.py + quality_gate.py — 统一路径解析

**文件：** `tain_agent/evolution/reporter.py`

- metrics snapshot 写入路径统一使用 `agent_workspace/{agent_name}/state/metrics_snapshots/`
- 消除多余的 `tain_agent/` 路径层级

**文件：** `tain_agent/evolution/quality_gate.py`

## 3. 链路 B：人格生命周期管理

### 3.1 当前断点

```
Agent 启动
  → _load_from_memory() → Memory 在 session 边界重置 → 空 ✗
  → 磁盘有 personality.json 但不读取 ✗ _load_from_disk 不存在

Agent 运行
  → personality_update → 写 T1 磁盘？→ _save_to_disk 路径硬编码 ✗
  → T2 运行时 trait 在 session 结束时丢弃 ✗
  → T3 叙事 milestone 不回写任何层 ✗

Agent 间
  → puzzle 和 sage 共享 agent_workspace/state/personality.json ✗
```

### 3.2 目标体验

```
Agent 启动
  → _load_from_memory() → 命中则返回（session 热数据）
  → _load_from_disk(agent_workspace) → 磁盘 fallback
  → 空 slate（仅首次启动）

Agent 运行
  → personality_update 同时写 T1（磁盘）和 T2（运行时）
  → session 结束时 T2 行为指纹自动回写 T1

Agent 间完全隔离
  → puzzle: agent_workspace/puzzle/state/personality.json
  → sage:  agent_workspace/sage/state/personality.json
```

### 3.3 具体改动

#### B-1：Personality.__init__ 增加 workspace_path 参数 + 磁盘 fallback

**文件：** `tain_agent/core/personality.py:58-77`

```python
def __init__(self, memory=None, workspace_path: str | Path | None = None):
    ...
    self._workspace_path = Path(workspace_path) if workspace_path else None
    self._load_from_memory()
    if self.is_empty() and self._workspace_path:
        self._load_from_disk()  # 新增：磁盘 fallback
```

**新增 `_load_from_disk` 方法：**
```python
def _load_from_disk(self) -> None:
    """Load personality from agent workspace disk storage."""
    if not self._workspace_path:
        return
    disk_path = self._workspace_path / "state" / "personality.json"
    if not disk_path.exists():
        return
    data = json.loads(disk_path.read_text(encoding="utf-8"))
    self._traits = data.get("traits", self._traits)
    self._version = data.get("version", self.VERSION)
    self._created_at = data.get("created_at", self._created_at)
    self._evolution_log = data.get("evolution_log", [])
```

#### B-2：_save_to_disk 使用 agent 专属路径

#### B-4：T2→T1 session 结束时回写

## 4. 链路 C：元认知与协作基础设施

### 4.1 当前断点

```
Agent 自省
  → 需要自建重型工具（self_pattern_miner, dream_loop 等）
  → 每次内省消耗数百行 conversation log 扫描
  → 框架已有 decision_log + milestones，但 Agent 访问不到结构化视图

Agent 间通信
  → send_message / check_messages 未作为工具注册
  → pull model 依赖 Agent 主动检查，导致 8+ 小时延迟

Agent 诊断
  → diagnose_framework() 结果停留在 Agent journal 中
  → 无法触发框架改进
```

### 4.2 目标体验

```
Agent 自省
  → get_self_profile() → 一个调用返回动作分布 + 工具排名 + trait 活跃度

Agent 间通信
  → send_message / check_messages 原生可用
  → Agent 启动时框架主动提示未读消息

Agent 诊断 → 框架改进
  → 结构化诊断输出写入 agent_workspace/{name}/diagnostics/
  → 开发者/CI 可读取并自动创建 issue
```

### 4.3 具体改动

#### C-1：框架级轻量内省 API — get_self_profile()

**文件：** `tain_agent/evolution/introspection.py`（约 60 行新文件）或直接在 bootstrap 中注册

提供三个子功能：
- `action_distribution(since_days: int)` — 从 decision_log 中统计各类动作计数
- `tool_usage_ranking(top_n: int)` — 从决策日志中聚合工具调用频率
- `trait_activity()` — 直接从 Personality 实例获取当前 T2 行为指纹

**文件：** `tain_agent/core/bootstrap.py` — 注册 `get_self_profile` 为框架工具

#### C-2：消息工具注册 + 启动通知

**文件：** `tain_agent/core/bootstrap.py`

- 注册 `send_message(to_agent: str, content: str)` 工具
- 注册 `check_messages()` 工具
- Agent 启动/唤醒流程中增加一次 `check_messages()`，有未读消息时主动注入提示

#### C-3：诊断反馈结构化输出

**新文件：** `tain_agent/evolution/diagnostic_feedback.py`（约 40 行）

Agent 可调用以输出结构化 JSON 诊断报告。存储到 `agent_workspace/{agent_name}/diagnostics/{timestamp}.json`。

### 4.4 验证标准

- `get_self_profile()` 返回的数据与 Agent 手动统计一致
- puzzle 发送消息后，sage 下次启动时收到未读消息提示
- Agent 调用诊断输出后，`agent_workspace/{name}/diagnostics/` 目录下存在对应 JSON 文件

## 5. 实施计划


## 6. 设计决策记录

- **链路组织优于模块组织：** 每条链路从 Agent 视角验证，确保"Agent 能用"而不只是"代码改了"
- **_load_from_disk 独立实现而非修改 _load_from_memory：** 避免破坏 Memory 层的单一职责。磁盘是 fallback，Memory 仍是 session 内优先数据源
- **T2→T1 单向回写而非双向同步：** T1（声明式）是 canonical source，T2 行为指纹作为补充信号单向流入，避免多写者冲突
- **sandbox 白名单暴露为工具而非配置文档：** Agent 在 forge 前可编程查询，避免文档腐化问题
